# 화성형 팔란티어 시스템 (Hwaseong Palantir System)

> **"흩어진 도시 데이터를 하나의 언어로 연결한다"**

---

## 왜 이 사업을 시작했는가

### 도시 데이터의 단절 문제

대한민국의 지방자치단체는 수십 개의 부서가 각자의 방식으로 데이터를 생산한다. 도로 개설 정보는 건설과에, 민원 현황은 민원실에, 환경 이슈는 환경과에, 복지 수요는 복지과에 각각 고립되어 있다. 뉴스와 SNS에서 쏟아지는 시민의 목소리는 어디에도 체계적으로 수집되지 않는다.

결과적으로 도시 관리자는 **파편화된 정보** 속에서 의사결정을 내려야 한다. 어느 지역에 어떤 민원이 집중되는지, 특정 개발사업이 지역사회에 어떤 반응을 일으키는지, 시민들이 실제로 체감하는 불편이 무엇인지를 **한눈에 파악할 수 있는 수단이 없다.**

이것이 이 프로젝트의 출발점이다.

---

## 온톨로지로 단절을 해결한다

### 데이터 연결의 핵심: 온톨로지

온톨로지(Ontology)란 서로 다른 출처의 데이터를 **공통된 개념 체계로 연결**하는 기술이다. 민원 데이터의 "동탄2동"과 뉴스 데이터의 "동탄신도시", 블로그의 "동탄역 근처"는 모두 같은 공간을 가리키지만 기존 시스템에서는 연결되지 않는다.

온톨로지 기반 도시 데이터 플랫폼은 이를 해결한다.

```
민원 데이터   ─┐
뉴스 데이터   ─┤──→ 온톨로지 엔진 ──→ 통합 공간 인텔리전스
블로그 데이터 ─┤         (장소·시간·주제 연결)
카페 데이터   ─┘
```

장소, 시간, 주제, 감성이라는 4개 축으로 모든 데이터를 정렬하면, **단절되어 있던 정보들이 의미 있는 패턴으로 연결**된다. 특정 지역의 교통 민원이 급증하는 시점과 신규 아파트 입주 시기가 겹친다는 사실, 특정 공원 인근에서 안전 관련 게시글이 반복적으로 올라온다는 패턴이 비로소 가시화된다.

## 팔란티어의 우수사례에서 배운다

### 팔란티어 테크놀로지스(Palantir Technologies)란

팔란티어는 2003년 미국에서 설립된 데이터 분석 플랫폼 기업이다. CIA, FBI, 미 국방부를 시작으로 현재는 영국 NHS(국민보건서비스), 우크라이나 국방부, 여러 지방정부에 이르기까지 **복잡한 다중 출처 데이터를 통합하여 의사결정을 지원**하는 시스템을 구축해왔다.

### 팔란티어의 핵심 성과

**영국 NHS 사례**
코로나19 팬데믹 당시 전국 병원의 병상 현황, 의약품 재고, 환자 이송 데이터를 실시간 통합하여 의료 자원을 최적 배분. 기존에는 수십 개 시스템에 흩어져 있던 데이터를 하나의 화면에서 운용함으로써 의사결정 속도를 획기적으로 단축시켰다.

**미국 지방정부 사례**
뉴올리언스, 로스앤젤레스 등 여러 도시에서 범죄 데이터, 인프라 노후화 데이터, 311 민원 데이터를 통합하여 자원 배분의 우선순위를 결정하는 데 활용하고 있다.

**공통된 성공 요인**
- 데이터 사일로(Data Silo) 해소
- 공간 데이터 기반 시각화
- AI를 활용한 패턴 탐지 및 예측
- 현장 담당자가 직접 사용할 수 있는 직관적 인터페이스

팔란티어가 증명한 것은 기술 자체가 아니다. **흩어진 데이터를 연결했을 때 의사결정의 질이 근본적으로 달라진다**는 것이다.

## 한국형 팔란티어 시스템을 만든다

### 왜 한국형인가

팔란티어는 강력하지만 한국 지방정부 환경에는 맞지 않는 부분이 있다. 도입 비용이 수십억 원에 달하고, 한국의 행정 데이터 구조와 법령 체계에 최적화되어 있지 않으며, 네이버·카카오 생태계에서 생산되는 한국형 시민 데이터를 처리하는 데 한계가 있다.

**화성형 팔란티어 시스템은 다르다.**

한국의 지방자치단체가 실제로 사용하는 데이터 구조에서 출발한다. 민원24, 새올행정시스템의 민원 데이터, 네이버 뉴스·블로그·카페에서 수집되는 시민 의견, 공공데이터포털의 행정 데이터를 하나의 지도 위에 통합한다.

### 화성시를 시작점으로 선택한 이유

화성시는 대한민국에서 가장 빠르게 성장하는 기초자치단체 중 하나다. 동탄신도시를 비롯한 대규모 택지개발, 삼성전자 반도체 클러스터, 급증하는 인구와 함께 도시 관리의 복잡성도 폭발적으로 증가하고 있다. 다양한 계층의 시민, 다양한 유형의 민원, 빠르게 변화하는 도시 구조는 데이터 통합 플랫폼의 필요성을 가장 명확하게 보여주는 환경이다.

화성시에서 작동하는 시스템은 대한민국 어느 도시에도 적용할 수 있다.

## 시스템 구조

### Beta v0.1 — 현재

```
네이버 뉴스·블로그·카페  ──→  수집 API
민원 엑셀 데이터         ──→  파일 파서
                               ↓
                        Claude AI 엔진
                     (장소 추출 + 분석)
                               ↓
                        네이버 지도 시각화
                               ↓
                        AI 분석 에이전트
                      (자연어 질의응답)
```

### 정식 버전 3-Tier — 목표

| 레이어 | 기술 스택 | 역할 |
|--------|-----------|------|
| **프론트엔드** | React + Vite | 대시보드, 지도, AI 채팅 UI |
| **API 서버** | Node.js + Express | 크롤러, AI 파이프라인, 온톨로지 엔진 |
| **데이터** | PostgreSQL + PostGIS + Redis | 공간 데이터 저장, 캐시, 스케줄링 |

- **실시간 데이터 수집** — 뉴스·블로그·카페·민원 자동 수집 및 지도 시각화

## 공간 데이터 체계

### 기준 공간 단위: 좌표 + 국가기초구역 (우편번호)

화성형 팔란티어 시스템은 **좌표(위경도)**와 **국가기초구역(우편번호 5자리)** 두 단위만을 공간 분석 기준으로 사용한다. 행정동·읍면동·시군구 단위 집계는 이 시스템의 범위 밖이다.

```
좌표 (위경도)  ←→  국가기초구역 (우편번호 5자리)
     │                      │
 지도 위 정확한          공간 분석·집계·
  위치 표시               히트맵 단위
```

공간 단위 체계상 위치 관계:
```
좌표  <  국가기초구역  <  행정동  <  시군구
         (이 시스템의 범위)
```

국가기초구역을 최대 공간 단위로 설정한 이유:
- 행정동보다 작고 실제 생활권에 가까운 단위로 민원 집중 지역 정밀 식별 가능
- 우편번호는 전국 표준 코드로 타 데이터셋과 조인이 용이
- 도로명주소 → 우편번호 변환이 API로 자동화 가능

### 데이터 소스별 공간 처리 방식

| 데이터 소스 | 원본 공간 정보 | 변환 방식 | 저장 단위 |
|-------------|---------------|-----------|----------|
| 민원 데이터 | 도로명주소 | 지오코딩 → 좌표 + 우편번호 추출 | 좌표 + 국가기초구역 |
| 뉴스·블로그 | 텍스트 내 장소명 | Claude NER → 지오코딩 | 좌표 + 국가기초구역 |
| 인프라 시설 | 좌표 | 역지오코딩 → 우편번호 매핑 | 좌표 + 국가기초구역 |

## 개인정보 비식별화 처리 체계

민원 데이터 연계 시 개인정보보호법 및 공공데이터 제공 및 이용 활성화에 관한 법률에 따라 수집 단계에서 비식별화·삭제 처리를 원칙으로 한다.

### 비식별화 처리 항목

| 항목 | 원본 | 처리 방식 | 시스템 저장값 |
|------|------|-----------|--------------|
| 민원인 성명 | 홍길동 | 완전 삭제 | null |
| 주민등록번호 | 800101-1****** | 완전 삭제 | null |
| 연락처 | 010-1234-5678 | 완전 삭제 | null |
| 상세주소 | 동탄역로 27, 101동 302호 | 도로명+건물번호까지만 | 동탄역로 27 |
| 민원 내용 | 이름·연락처 포함 서술 | Claude API 마스킹 처리 | 개인정보 제거 텍스트 |
| 처리 담당자 | 김○○ 주무관 | 부서명만 유지 | 도로관리과 |

### 비식별화 처리 흐름

```
민원 원본 데이터 (새올행정시스템)
    ↓
개인정보 필드 자동 삭제 (ETL 파이프라인)
    ↓
Claude API — 민원 내용 내 잔존 개인정보 마스킹
    ↓
상세주소 → 도로명+건물번호 truncation
    ↓
국가기초구역 코드로 공간 집계
    ↓
화성형 팔란티어 시스템 DB 저장
```

## 데이터 거버넌스 체계

화성형 팔란티어 시스템의 데이터 거버넌스는 **도시데이터 정책협의체**와 **실행협의체** 이원 체계로 운영한다.

### 거버넌스 구조

```
┌─────────────────────────────────────────────────┐
│              정책협의체 (Policy Layer)             │
│  위원장: 제1부시장 / 부위원장: 제2부시장            │
│  구성: 실장·국장·소장급 (30명 이내)               │
│                                                  │
│  심의·조정 사항                                   │
│  • 도시상태 정의                                  │
│  • 부서별 문제상태 정의 및 기준                   │
│  • 핵심 데이터 지정 및 관리 방향                  │
└────────────────────┬────────────────────────────┘
                     │ 방침 하달 / 이행현황 보고
┌────────────────────▼────────────────────────────┐
│              실행협의체 (Execution Layer)          │
│  위원장: 과장급 / 부위원장: 팀장급                │
│  구성: 과장·팀장·정보시스템·공공데이터 담당자      │
│                                                  │
│  협의·실행 사항                                   │
│  • 데이터 표준 적용 및 메타데이터 관리             │
│  • 데이터 품질 점검 및 오류·중복 개선              │
│  • DB 연계, 부서 간 데이터 공유                   │
│  • 공공데이터 개방 및 활용 확대                   │
│  • 도시상태 기반 데이터 분석 및 정책 활용 지원     │
└────────────────────┬────────────────────────────┘
                     │ 운영 지침
┌────────────────────▼────────────────────────────┐
│         화성형 팔란티어 시스템 (기술 레이어)        │
│  국가기초구역 공간 온톨로지                        │
│  비식별화 ETL 파이프라인                          │
│  AI 분석 에이전트 (Claude API)                    │
│  네이버 지도 시각화 대시보드                       │
└─────────────────────────────────────────────────┘
```

### 거버넌스와 시스템의 연결

| 거버넌스 사항 | 시스템 구현 항목 |
|--------------|----------------|
| 도시상태 정의 | 데이터 수집 키워드·카테고리 분류 체계 |
